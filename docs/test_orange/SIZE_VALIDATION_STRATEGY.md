
# Три уровня стратегии валидации UI-состояния

## 1. Проверка по CSS-классу
Этот метод проверяет логическое состояние компонента, а не его визуальные параметры напрямую.

### Пример кода
```python
sidepanel = page.locator("aside")
expect(sidepanel).not_to_have_class(re.compile("toggled"))
sidepanel.get_by_role("button").click()
expect(sidepanel).to_have_class(re.compile("toggled"))
````

* **Что валидируется:** UI-state (например, переход sidebar из `expanded` в `collapsed`).
* **Плюсы:** Самый надежный и быстрый способ. Не зависит от layout, DPI или дробных пикселей. Минимально подвержен «флаканью» (flaky-поведению). Является предпочтительным методом для production.
* **Минусы:** Работает только при наличии предсказуемых CSS-классов.
* **Когда использовать:** Если UI управляется CSS-состояниями (классы типа `collapsed`, `toggled`). Это **первый выбор** в реальных проектах.

---

## 2. Проверка по CSS-свойствам

Метод проверяет вычисленные (computed) CSS-значения элемента через `getComputedStyle(el)`.

### Пример кода

```python
expect(sidepanel).to_have_css("width", "256px")
sidepanel.get_by_role("button").click()
# Рекомендуется использование регулярного выражения для обработки дробных пикселей
expect(sidepanel).to_have_css("width", re.compile(r"83(\.\d+)?px"))
```

* **Что валидируется:** Визуальный стиль элемента (например, изменение ширины с 256px до 83px).
* **Плюсы:** Хороший компромисс между точностью и стабильностью. Благодаря `expect()` поддерживается автоматический повтор (auto-retry).
* **Минусы:** Риск из-за дробных пикселей (зависит от zoom/DPI/OS). Также CSS-значение (например, `width: auto`) не всегда совпадает с реальной геометрией.
* **Когда использовать:** Когда важен именно стиль, но отсутствуют стабильные классы. Это **второй по приоритету** метод.

---

## 3. Проверка через DOM-геометрию

Проверка реальной геометрии элемента в макете (layout) с использованием функции `wait_for_size_stable`, которая учитывает стабилизацию размеров элемента.

### Пример кода

```python
sidepanel = page.get_by_role("navigation", name="Sidepanel")
sidepanel.get_by_role("button").click()

# Ждём стабилизации размера боковой панели
size = wait_for_size_stable(sidepanel)

print(f"Width: {size['width']}, Height: {size['height']}")
```

* **Что валидируется:** Настоящий размер элемента на экране (layout width и height), а не CSS-значение.
* **Плюсы:** Самый точный метод. Работает с flex/grid, анимациями и динамически вычисляемой шириной/высотой.
* **Минусы:** Медленнее предыдущих методов, ресурсоёмкий, может быть подвержен «флаканью» на сложных макетах.
* **Когда использовать:** Когда CSS-значения нестабильны, используются анимации или нужен фактический размер в динамическом макете. Это **инженерный fallback-метод**.

---

## Сравнение методов

| Метод               | Проверяет       | Скорость    | Надежность       | Когда использовать        |
| :------------------ | :-------------- | :---------- | :--------------- | :------------------------ |
| **Class**           | UI State        | 🟢 Быстро   | 🟢 Очень высокая | Production (Приоритет №1) |
| **CSS**             | Computed Style  | 🟡 Средне   | 🟡 Хорошая       | Когда нет классов         |
| **Evaluate/Layout** | Layout Geometry | 🔴 Медленно | 🟡 Средняя       | Сложный layout / Анимации |

---

### Инженерная рекомендация

Следуйте строгому порядку выбора метода: **Class → CSS → Geometry**. Никогда не используйте их в обратном порядке.
